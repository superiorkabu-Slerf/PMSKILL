# Dynamic Questioning Guide

不要按行业类别去背题库。先识别“判断这件事是否值得做，还缺什么关键信息”，再生成问题。

## 零、大词收窄规则

如果用户说的是“做一个社区”“做一个平台”“做一个生态”“做一个给所有人用的产品”，不要顺着这些词往下分析。

必须先把需求压缩成：

- 谁是最核心的第一批用户
- 他们在哪个具体场景下遇到问题
- 他们最先会发生的一个关键动作是什么

如果这三件事还说不清，就继续追问，不进入市场调研和结论阶段。

## 一、先做信息缺口扫描

每次开始分析前，快速检查以下九个维度：

1. 问题：问题是什么，是否足够具体
2. 用户：谁最痛，谁最早会采用
3. 场景：问题在什么场景下被触发
4. 频率：多久发生一次，损失有多大
5. 替代：用户现在怎么解决
6. 付费：谁付钱，预算从哪里来
7. 优势：为什么你有资格切入
8. 约束：合规、供应、技术、平台限制
9. 验证：用什么信号证明方向成立

先找“最不确定但最关键”的 3-5 个缺口，再围绕缺口追问。

## 二、问题生成规则

- 每个问题必须对应一个缺口，不问泛泛而谈的问题
- 优先问会改变判断方向的问题
- 一轮控制在 3-5 个问题
- 问题要具体到场景，不要只问观点
- 允许加入帮助思考的例子，但不要给预设答案

## 三、常见问题骨架

以下是问题骨架，不是固定题库。应根据用户语境改写。

### 1. 问题真实性

- 这个问题最近一次发生在什么场景
- 当问题发生时，用户损失的是时间、金钱、机会还是体验
- 如果不解决，用户会继续忍受，还是会主动寻找替代方案

### 2. 用户优先级

- 哪一类用户最先会因为这个问题采取行动
- 是使用者痛，还是决策者痛，还是付费者痛
- 谁最可能成为第一批愿意尝试的人

### 3. 频率和强度

- 这是每天、每周还是偶发问题
- 问题造成的损失能不能被量化
- 问题是“烦”，还是“必须解决”

### 4. 替代方案

- 用户今天不用你，会怎么做
- 他们为什么继续忍受现有方案
- 现有方案最大的不满点是什么

### 5. 付费与商业化

- 谁会为这个方案买单
- 用户现在是否已经在为类似结果付费
- 你的方案是节省成本、增加收入，还是降低风险

### 6. 进入壁垒

- 这个方向最难的不是功能，而是什么
- 为什么别人还没有把这件事做好
- 如果大厂复制，用户为什么不直接转过去

### 7. 验证计划

- 哪个假设最可能错
- 用最小代价验证，应该先测什么
- 什么结果出现时，应立即停止继续投入

## 四、追问顺序建议

通常优先级如下：

1. 问题是否真实
2. 用户是否明确
3. 替代方案是否存在明显不足
4. 是否有人愿意为结果付费
5. 当前风险能否被快速验证

只有当前面几项初步成立，再去展开更细的市场和竞争分析。
