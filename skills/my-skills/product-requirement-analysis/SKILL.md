# Product Requirement Analysis

用于把模糊的产品想法、需求方向或功能机会，转化为一份可讨论、可验证、可挑战的分析结论。

这不是一个“帮用户把想法包装得更好看”的 skill。目标是识别真实机会、暴露关键风险，并给出下一步最有价值的验证动作。

## 何时使用

在以下情况触发：

- 用户只有一个初步想法，希望判断值不值得做
- 用户提出一个新需求，希望分析用户价值、市场空间与优先级
- 用户想比较多个方向，收敛到更值得验证的一个
- 用户需要一份比普通头脑风暴更严谨的产品分析文档

不适用于：

- 已经明确要实现，且只需要写 PRD
- 纯 UI 文案、页面结构或交互优化
- 明确是技术方案设计而非需求分析

## 核心原则

- 先识别信息缺口，再提问，不要套行业题库
- 问题少而准，每轮只问最影响判断的 3-5 个问题
- 区分事实、推断、假设，不要混写
- 先验证“是否值得做”，再展开“怎么做”
- 默认保持怀疑，不替用户自动补全关键商业结论
- 结论必须带置信度，并指出还缺什么证据
- 约束运行路径，不约束具体措辞、文档形态和表达风格

## 运行路径约束

本 skill 不提供固定话术、固定问题集或固定输出模板，但必须遵守以下路径顺序：

1. 先收窄需求对象
2. 再进行动态追问
3. 再做外部验证
4. 再选择必要的分析框架
5. 再给阶段性判断并做压力测试
6. 最后给优化建议或下一步验证动作

任何一步都不能跳过前置条件：

- 如果需求对象仍然过大、过虚，不能直接开始调研
- 如果关键信息仍然不足，不能直接给确定性结论
- 如果没有经过反方找茬，不能直接输出“建议继续”

允许不同模型、不同场景自由发挥，但必须满足以下最小约束：

- 至少有一轮追问
- 至少有一轮外部信息验证
- 至少有一次明确结论
- 至少有一次找茬
- 至少有一轮优化或收敛建议

## 工作流

### 1. 识别分析对象

先判断用户在分析什么：

- 全新产品方向
- 现有产品的新需求
- 面向某类用户的新场景
- AI/自动化带来的新机会
- 商业化或增长相关方向

同时识别当前成熟度：

- 只有模糊想法
- 已有明确目标用户
- 已有初步方案
- 已有部分验证或竞品认知

如果输入非常模糊，不要直接进入完整框架，先进入动态追问阶段。

如果用户提出的是“社区”“平台”“生态”“一站式”“面向所有人”这类大词，必须先按 [references/dynamic-questioning.md](references/dynamic-questioning.md) 中的“大词收窄规则”收窄后再继续。

如果用户在流程中实质性改变了分析对象，例如从“宠物社区”切换到“宠物保险”或从“新项目”切换到“功能优化”，必须回到本步骤重新识别，而不是在旧上下文中硬拐。

### 2. 动态追问，不使用静态分类题库

必须先评估当前信息缺口，再生成问题。优先检查以下维度：

- 问题是否真实存在
- 用户是谁，谁最痛
- 发生频率和触发场景是什么
- 当前替代方案是什么
- 用户为何不满意现有方案
- 谁会付费，为什么会付费
- 供给侧难度、资源门槛和合规约束是什么
- 团队或项目是否拥有独特优势
- 成功失败如何衡量

只问最关键的 3-5 个问题，遵守以下规则：

- 每个问题都要服务于一个明确的不确定点
- 每个问题默认提供至少 3 个强相关选项，外加 1 个“其他/自行补充”入口
- 选项的作用是帮助用户聚焦，不是替用户决定答案
- 如果用户明显不熟悉该领域，可用选项帮助其快速判断，但仍允许用户自由补充
- 优先问那些一旦答案不同，就会推翻整个方向的问题
- 如果已有信息足够，不要为了凑流程继续提问

生成选项时，必须遵守以下约束：

- 选项必须紧贴当前需求语境，不能给泛泛的行业分类
- 选项之间要尽量互斥，避免语义重叠
- 选项数量控制在 3-4 个，防止用户失焦
- 必须保留用户自主输入空间，避免把真实需求限制在选项里
- 如果用户选择“其他/自行补充”，应继续围绕其补充内容追问，而不是强行拉回既有选项

追问的停止条件、轮次上限和方向重置规则，统一参考 [references/dynamic-questioning.md](references/dynamic-questioning.md)。

### 3. 证据化验证

在做市场、竞品和用户判断时，优先调用搜索或可用资料进行外部验证。

最低要求：

- 至少确认 3 类外部信息：市场信号、替代方案、真实用户反馈
- 竞品通常看 3-7 个，宁可少而深，不要机械凑 10 个
- 能给出时间和来源时，必须给出时间和来源
- 找不到可靠证据时，要明确标注为“待验证假设”，不能写成事实
- 在进入正式调研前，可询问用户是否有补充材料可一并纳入分析，补充材料不是强制项

重点验证：

- 这个问题是否高频或高损失
- 用户是否已有替代方案，且替代方案是否足够差
- 市场是否已经拥挤到难以切入
- 需求是否存在支付意愿或预算归属
- 机会是否受政策、平台、数据或供应链限制

如果用户已有额外资料，应优先纳入判断，例如：

- 用户调研报告
- 流量或转化数据
- 历史功能数据
- 销售、客服或运营反馈
- 访谈纪要、问卷结果、社区反馈整理

如果用户没有补充材料，也不要阻塞流程，但应明确说明：

- 当前已有证据足以支撑哪些判断
- 还缺哪些材料会显著提升判断质量
- 最值得补充的 1-3 份材料是什么

证据质量标准见 [references/evidence-rubric.md](references/evidence-rubric.md)。

### 4. 选择必要的分析框架

框架不是必选项，只有在能提升判断质量时才使用。

优先按以下条件选择：

- 涉及政策、平台规则或宏观环境明显影响时，用 PEST
- 存在清晰竞争格局、需要判断切入点时，用 SWOT 或竞品对比矩阵
- 同时比较多个方向或多个切入方案时，用加权评分矩阵
- 商业模式、收入来源或关键资源不清时，用商业画布或轻量商业闭环
- 对单一功能需求，优先分析用户价值、替代方案、触发频率、实现代价和验证路径，不必强行套大框架

如果无法说明“为什么要用这个框架”，就不要使用。

### 5. 阶段性判断、压力测试与重新对齐

完成初稿后，先给出一个阶段性判断，再切换到“反方模式”，主动攻击这个方向。

至少从以下角度挑战一次：

- 用户会不会只说痛，不付费
- 需求是不是低频、伪刚需或一次性需求
- 获客成本是否会吃掉价值
- 团队是否缺少关键资源或行业进入门槛
- 大厂、平台或更强竞品是否能快速复制
- 合规、数据、交付链路中是否有致命约束

如果一个问题可能直接推翻项目，必须明确标成“生死线问题”。

需要展开找茬时，参考 [references/pressure-test.md](references/pressure-test.md)。

把最关键的风险、空白和待确认假设抛回给用户，不要把所有问题都摊开。

优先确认：

- 最可能误判的用户假设
- 最薄弱的商业化假设
- 最缺乏证据支撑的竞争判断
- 最可能拖死项目的资源或合规问题

如果用户给出的新信息足以改变方向，应允许结论转向，而不是强行维持原方案。

### 6. 输出最终分析结论

最终输出应兼具“判断”与“行动建议”，但不强制固定格式。

无论输出是对话式、纪要式、报告式还是表格式，至少要覆盖：

- 当前理解的需求对象
- 追问后补齐的关键信息
- 调研得到的外部信号
- 当前结论
- 主要风险或生死线问题
- 优化建议或下一步验证动作
- 分析日期、分析版本和信息截止日期

## 输出要求

### 必须包含

- 清晰结论，而不是只有分析过程
- 哪些内容是事实，哪些是推断
- 最大的 3 个风险
- 当前最值得做的下一步验证动作
- 对该方向的总体判断：`值得继续`、`谨慎推进`、`暂不建议`

### 推荐包含

- 1 句价值主张
- 用户细分与首个切入场景
- 主要竞品或替代方案对比
- 最小可验证 MVP
- 成功指标与失败信号

### 禁止事项

- 在缺乏证据时给出确定性口吻
- 机械罗列框架名词但没有结论
- 把用户表达的愿望直接当成用户真实需求
- 把“AI 能做”误写成“用户愿意买”

## 快速判断标准

如果需要快速判断一个方向是否值得继续，至少回答这 5 个问题：

1. 是否存在明确且可描述的高痛场景
2. 是否有一类用户比其他人更急、更痛、更愿意尝试
3. 当前替代方案是否明显不够好
4. 该方向是否存在可进入的商业化路径
5. 当前最大风险是否能在低成本实验中尽快验证

如果其中 3 个以上无法回答，不要假装分析完成，应先回到追问和验证阶段。
